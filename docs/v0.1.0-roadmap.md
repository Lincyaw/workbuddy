# v0.1.0 实施路线图

## 目标

单机跑通完整闭环：Issue 创建 → Agent 执行 → Agent 改 label → 下一个 Agent 触发 → 直到完成或超过重试上限。

## 架构模式

v0.1.0 是**单机合体**模式：Coordinator + Worker 在同一进程内通过 Go channel 通信。

```
workbuddy serve (单进程)
  ├── Coordinator: Poller → StateMachine → TaskRouter ──channel──┐
  ├── Coordinator: Reporter ← result channel                     │
  ├── Coordinator: SQLite Store                                  │
  ├── Coordinator: Event Log                                     │
  └── Worker: TaskExecutor ← channel ← ClaudeCodeLauncher ──────┘
```

```

```

**测试场景**:
1. **Happy path (Agent-as-Router 闭环)**:
   - 创建 Issue + `type:feature` label
   - Poller 检测 → StateMachine 匹配 workflow → 进入 triage → developing
   - TaskRouter 派发给内嵌 Worker → Worker 执行 agent（mock 命令，执行 gh issue edit 改 label）
   - Poller 检测到新 label → 下一个 state → 下一个 agent
   - 最终到达 done → Issue 关闭

2. **Retry loop**:
   - Agent 执行完设置回退 label（如 status:developing）
   - CycleTracker 计数正确
   - 第 4 次回退 → 自动转 failed + needs-human

3. **Retry limit → failed**:
   - 模拟 3 次回退 → 验证自动转 failed

4. **Config error**:
   - Workflow 引用不存在的 agent → 启动报错

## 风险与缓解

| 风险 | 影响 | 缓解 |
|------|------|------|
| Agent（Claude Code）不按预期改 label | 流程卡住 | Coordinator 设置超时——agent 执行完但 label 没变化 > N 分钟，标记为 stuck |
| SQLite 纯 Go 库性能 | 写入瓶颈 | WAL 模式 + batch writes；workbuddy 的写入量极低，不太可能成为瓶颈 |
| Markdown frontmatter 解析 | 复杂度 | 用简单的 --- 分割，不引入 goldmark |
| 多 goroutine 竞争 | 运行时 bug | 关键数据用 sync.Mutex；`go test -race` |
| Agent 执行时间不可预测 | shutdown 慢 | 60s 等待上限，超时强制终止 |
| gh CLI 在不同环境行为差异 | 测试不可靠 | 核心逻辑用 GHExecutor mock；E2E 测试手动运行 |
